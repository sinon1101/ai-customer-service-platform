# AI 智能客服平台 — 改造设计方案

> 本文档记录将 **hm-dianping(黑马点评)** 改造为 **多租户 AI 智能客服 SaaS 平台** 的设计方案。
> 核心思路:**保留点评项目沉淀的 Redis/中间件基础设施(鉴权、缓存、分布式锁、限流、ID 生成、Stream 等),把业务主体从「本地生活点评」换成「生产级 AI 客服」**,用一套高并发治理体系驯服「又慢又贵」的大模型后端。

---

## 一、一句话定位

> 一个**多租户**的 **AI 智能客服 SaaS 平台**:企业上传自己的知识库,平台基于 **RAG** 提供 7×24 智能问答,答不了自动转人工坐席;底层用 **Redis + RocketMQ** 构建完整的高并发治理体系(缓存、限流、削峰、降级、熔断、隔离),让慢而贵的 LLM 后端扛住高并发流量。

**项目的灵魂 = 高并发治理**,AI 只是「面子」,工程能力才是「里子」。

---

## 二、为什么做这个(价值主张)

做 AI 应用的人分两类:
- **只会调 API 的「Prompt 工程师」**:`chat()` 一把梭,demo 炫,一上量就崩(没缓存、没限流、LLM 一抖整个服务挂)。
- **能把 AI 跑成生产级服务的后端工程师**:稀缺,也是企业真正缺的。

本项目的价值不在「会接大模型」(谁都会),而在用 **Redis 那套硬核基础设施解决 AI 应用真正的工程痛点**:

| 企业真正头疼的 | 本项目的解法 |
|---|---|
| LLM 调用太贵(烧 token) | **语义缓存**:相似问题命中缓存直接返回,不调模型 |
| 大模型会抖 / 超时 / 限速 | **限流 + 降级 + 熔断**:护住配额,LLM 挂了兜底 FAQ |
| 知识库文档处理慢 | **MQ 异步管道**:上传 → 异步切片+向量化,不阻塞用户 |
| 多客户数据隔离 | **多租户**逻辑隔离(tenant_id + Redis key 前缀) |
| 答不了要转人工 | **坐席抢工单** + 工单消息流 |

## 三、高并发设计(项目核心卖点)⭐

高并发有两副面孔,本项目**两者都覆盖**:

### A. 高并发「流量治理 + 慢后端保护」(主菜)

LLM 又慢(秒级)又贵(烧 token),是高并发治理最经典的靶子。比「再写一个秒杀」更新、更难、更值钱。

| 能力 | 设计 | 复用的点评经验 |
|---|---|---|
| **语义缓存(抗高并发读)** | 热门问题 = 热 key,命中缓存直接返回不打模型;覆盖**缓存击穿/穿透/雪崩**三件套 | `CacheClient`(逻辑过期 + 互斥锁、空值缓存)进化为语义缓存 |
| **限流** | 按租户/用户**令牌桶 + 滑动窗口**,护住 LLM 配额 | Redis 计数 + Lua 原子 |
| **削峰** | 突发洪峰 → RocketMQ 排队,消费端按 LLM 能力慢慢消费 | Redis Stream 经验迁移到 RocketMQ |
| **降级 / 熔断** | LLM 超时/不可用 → 降级到 FAQ / 缓存兜底,不让链路崩 | 新增(Sentinel / Resilience4j) |
| **资源隔离** | 多租户下,某租户流量暴涨不拖垮别人(信号量/线程池隔离 + 按租户配额) | 新增 |

### B. 高并发「写争抢」(配菜)— 坐席抢工单

机器人答不了 → 转人工 → **多个坐席「抢」同一个待接入会话**。这是 **seckill 的同构问题**:多客户端争抢同一资源,保证只有一个抢到、不超卖。

- **Redis 原子操作 / Lua / 分布式锁原封不动复用**(`SimpleRedisLock`、Redisson `RLock`)。
- 抢单成功 → 发 MQ 通知 → 建立会话路由。

> **总结:点评是「抢库存」的高并发;本项目是「治理 + 抢工单」的高并发。Redis 依然是绝对主角,故事更新更硬核。**

## 四、技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 框架 | Spring Boot 2.3.12 / Java 8 | 沿用,基础设施直接复用 |
| ORM | MyBatis-Plus | 沿用 |
| **缓存 + 向量库** | **Redis Stack(Docker)** | 一套 Redis 既当缓存又当向量库(RediSearch 向量检索) |
| **消息队列** | **RocketMQ** | 异步文档管道、削峰、工单流、延迟/重试 |
| **大模型** | **阿里百炼(通义)** | 有免费额度、合规;通过 **Spring AI** 统一抽象,换模型不改业务 |
| 分布式锁 | Redisson / 自研 Lua 锁 | 沿用,用于坐席抢工单 |
| 限流/熔断 | Redis + Lua / Sentinel / Resilience4j | 高并发治理 |
| 前端 | 沿用 nginx + 静态页(后续可换) | — |

### 关于 Redis 版本(重要)

- 向量检索靠 **RediSearch 模块**,普通 `redis-server` 不带。
- **RediSearch 2.4(2022)** 起支持向量检索;**Redis Stack 6.2.x / 7.x** 打包好开箱即用;**Redis 8(2025)** 把向量能力合进了核心开源版(原生 `Vector Set`)。
- ⚠️ **当前环境的 `Redis-x64-3.2.100` 是 2016 年微软停更的移植版,不支持向量,必须更换。**
- Windows 推荐 Docker 部署:
  ```bash
  docker run -d --name redis-stack -p 6379:6379 -p 8001:8001 redis/redis-stack:latest
  # 6379 = Redis;8001 = RedisInsight 可视化界面
  ```

## 五、整体架构

```
                          ┌─────────────────────────────────────┐
   租户后台(企业)         │  知识库管理 / 坐席工作台 / 数据看板   │
                          └──────────────┬──────────────────────┘
                                         │ 上传文档(PDF/Word/MD/网页)
                                         ▼
                                  发 RocketMQ(削峰,慢任务)
                                         │
                          ┌──────────────▼──────────────┐
                          │  异步管道:切片 → 向量化      │──► Redis Stack(向量库)
                          └──────────────────────────────┘

   访客提问
      │
      ├─► 多租户鉴权(token 双拦截器,带 tenant_id)        ← Redis
      ├─► 限流(按租户/IP 令牌桶 + 滑动窗口)               ← Redis + Lua
      ├─► 语义缓存:相似问命中直接返回(省 token)          ← Redis(向量相似度)
      │        │未命中
      │        ▼
      ├─► 向量召回 Top-K 知识片段                          ← Redis Stack
      ├─► 拼 Prompt → 阿里百炼 LLM → SSE 流式返回
      │        │(超时/失败)
      │        ▼ 降级/熔断 → FAQ 兜底
      ├─► 多轮对话上下文                                   ← Redis(Hash/List)
      ├─► 答不了 → 转人工:多坐席抢工单                     ← Redis 原子/分布式锁
      │        └─► 工单消息流                              ← RocketMQ / Redis Stream
      └─► 会话/满意度统计                                  ← Redis(可选 Kafka 行为流)
```

## 六、核心模块设计

### 1. 多租户体系
- **定位**:一套部署服务 N 家企业,各自知识库/数据/配置完全隔离。
- **隔离方式**:**逻辑隔离**(共享库)——每张表加 `tenant_id` 字段,Redis key 统一加租户前缀(如 `t:{tenantId}:kb:...`)。正好考验 key 设计能力,与点评「key 前缀实战」一脉相承。
- 租户注册 / 配置(模型 Key、配额、限流阈值)/ 计量。

### 2. 知识库管理 + 异步向量化管道
- 上传文档(PDF/Word/Markdown/网页 URL)。
- 上传即返回,**发 RocketMQ** → 消费端**切片(chunk)→ 调百炼 embedding → 写 Redis Stack 向量索引**。
- 失败重试、死信队列(RocketMQ 原生能力);处理进度回写,前端轮询/推送。

### 3. RAG 问答引擎
- 提问 → embedding → 向量召回 Top-K → 拼 Prompt → LLM → **SSE 流式返回**。
- 多轮对话上下文存 Redis(限定窗口长度,控制 token)。
- 引用溯源:回答附带命中的知识片段来源。

### 4. 语义缓存(省钱 + 抗并发核心)
- 问题向量化后,先在「缓存向量集」里找相似度超阈值的历史问答,命中直接返回。
- 覆盖缓存穿透(空值缓存)、击穿(互斥锁/逻辑过期)、雪崩(TTL 随机)。

### 5. 人工坐席系统
- 机器人无法回答 / 用户主动转人工 → 生成工单进入待接入池。
- **多坐席抢工单**(高并发写争抢,Redis 原子/分布式锁)。
- 坐席与访客实时会话(WebSocket + Redis Stream/RocketMQ 做消息流)。

### 6. 流量治理(横切)
- 限流、降级、熔断、隔离,作为统一的网关/切面能力,贯穿所有入口。

### 7. 数据看板(可选进阶)
- 会话量、缓存命中率、token 消耗、满意度、坐席效率。
- 高级版:用户行为 → Kafka 流 → 实时分析(只有真要做大规模分析时才上 Kafka)。

## 七、可直接复用的点评基础设施

| 点评里的组件 | 在新项目里的用途 |
|---|---|
| Token 双拦截器(`RefreshTokenIntercepter` + `LoginIntercepter`) | 改造为**多租户鉴权**,ThreadLocal 里带 `tenant_id` |
| `CacheClient`(逻辑过期 + 互斥锁 + 空值缓存) | 进化为**语义缓存**,缓存三件套直接复用 |
| `RedisIdWorker` | 会话 ID、工单 ID 生成 |
| `SimpleRedisLock` / Redisson `RLock` | **坐席抢工单**的原子争抢 |
| Redis Stream 经验 | 平滑迁移到 RocketMQ;工单消息流 |
| `RedisConstants`(key 前缀集中管理) | 升级为**带租户前缀**的 key 规范 |
| `Result` / DTO / 分层结构 | 沿用 |

## 八、开发路线图(建议里程碑)

1. **M0 环境**:Docker 起 Redis Stack;搭 RocketMQ;接通阿里百炼(Spring AI 跑通一次对话)。
2. **M1 骨架**:多租户模型 + 鉴权改造 + 基础 CRUD(租户/知识库)。
3. **M2 知识库管道**:文档上传 → RocketMQ → 异步切片向量化 → 写入 Redis 向量索引。
4. **M3 RAG 问答**:向量召回 + LLM + SSE 流式返回 + 多轮上下文。
5. **M4 语义缓存**:相似问命中缓存 + 缓存三件套。
6. **M5 高并发治理**:限流 + 削峰 + 降级熔断 + 隔离。
7. **M6 人工坐席**:转人工 + 坐席抢工单 + 实时会话。
8. **M7 看板/打磨**:统计看板、压测验证高并发指标。

> 建议每个里程碑可独立演示,简历上每条都是亮点。

## 十、关键设计决策记录(便于回顾)

| 决策点 | 选择 | 理由 |
|---|---|---|
| 应用主体 | 多租户 AI 智能客服平台 | 复用点评基础设施,AI 是真主体,企业刚需 |
| 向量库 | Redis Stack | 一套 Redis 兼做缓存 + 向量库,与「Redis 实战」主题统一 |
| 大模型 | 阿里百炼(Spring AI 抽象) | 免费额度、合规、生态契合 |
| MQ | RocketMQ | 任务型负载契合,延迟/重试原生,运维轻,阿里系生态统一 |
| 租户模式 | 多租户 · 逻辑隔离 | 像真 SaaS,工作量可控,考验 key 设计 |
| 高并发定位 | 流量治理(主)+ 坐席抢工单(配) | 比第 1001 个秒杀更新、更硬核 |
