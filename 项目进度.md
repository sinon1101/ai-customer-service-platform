## 一、项目是什么

把 **hm-dianping(黑马点评)** 魔改为 **多租户 AI 智能客服 SaaS 平台**:
保留点评沉淀的 Redis/中间件基础设施(鉴权、缓存、分布式锁、ID 生成、Stream 等),
业务主体换成「生产级 AI 客服」,用一套**高并发治理体系**(缓存/限流/削峰/降级/熔断/隔离)驯服又慢又贵的 LLM。

- **技术栈**:Spring Boot 2.3.12 / Java 8 / MyBatis-Plus / Redis Stack(缓存+向量库) / RocketMQ / 阿里百炼(Spring AI) / 多租户逻辑隔离。
- **项目灵魂**:高并发治理。AI 是面子,工程能力是里子。

### 2026-06-16 — M6:人工坐席(转人工 + 坐席抢工单 + 实时会话)
- **定位**:补齐「机器人答不了 → 转人工」的闭环。本里程碑的高并发主角从「读治理」转到**「写争抢」**——多坐席抢同一张工单,是点评秒杀的同构问题(多客户端争抢一份资源,保证只有一个抢到、不超卖)。
- **角色设定**:访客 = 登录的 `sys_user`(发起转人工者);坐席 = 同租户 `role=AGENT`(ADMIN 在 demo 里可兼任)。WebSocket 鉴权与工单归属全复用现有多租户链路。
- **请求闭环**:`/chat` 降级/答不了或用户主动 →(访客)`POST /ticket/transfer` 建 **WAITING** 工单进待接入池 →(坐席)`GET /ticket/pending` 看池 + `POST /ticket/{id}/claim` **抢单** → 抢到者与访客经 **WebSocket** 实时会话 → `POST /ticket/{id}/close` 结束。

WHERE id=? AND tenant_id=? ### 2026-06-15 — M4:语义缓存(相似问命中 + 缓存三件套)
- **定位**:在 M3 召回链路<b>之前</b>加一层语义缓存短路。问题向量化后先在「缓存向量集」按高阈值找相似历史问答,命中直接返回(省一次 LLM 调用、抗高并发读);未命中才走 RAG,并把结果回填缓存。
- **独立缓存向量索引**:`config/VectorStoreConfig` 新增第二个 `RedisVectorStore` bean `semanticCacheStore`(索引 `aics-cache-index` / 前缀 `aics:cache:`,复用同一条 `JedisPooled`),与知识库索引 `aics-kb-index` 分库 —— 这样能<b>单独给缓存项设 TTL</b>(雪崩抖动/空值短 TTL)。两个同类型 bean 靠 `@Resource` 按字段名注入区分(`vectorStore` vs `semanticCacheStore`)。
- **命中阈值 0.92(与召回的 0.3 分离)**:缓存必须比召回严得多,否则会把「相近但不同」的问题误判命中、答错。实测换措辞(「用了10天还能退款吗」↔「东西买了十天了还能退吗」)能命中,语义而非字面匹配。
- **缓存三件套**(封装在 `cache/SemanticCache`):
  - **穿透(空值缓存语义版)**:知识库找不到答案的「未命中型」回答也入缓存,但只给很短 TTL(`NULL_ANSWER_TTL_SECONDS=120s`),让相似无解提问短路、不反复打 LLM。<b>关键坑</b>:召回阈值 0.3 太松,无关问题也会召回到 chunk,`docs 非空 ≠ 答出来了`;改用<b>答案文本是否含兜底语</b>(`ChatConstants.NOT_FOUND_MARKER="知识库中暂未找到"`)判定 answered,决定长/短 TTL。
  - **击穿(互斥锁)**:未命中时 `setIfAbsent` 抢重建锁(key 维度 = 租户+kbId+问题 md5),只放一个线程打 LLM 并回填;<b>等待者廉价轮询锁释放</b>(只 GET 锁键、不重复 embedding),持锁者算完再做一次查找读其结果。`LOCK_WAIT_MAX_MILLIS=8s` 须覆盖一次 LLM 耗时,否则等待者提前放弃各自打 LLM。实测清缓存后 5 并发同问 → <b>仅 1 次 LLM 调用</b>(cached=false)+ 4 个等锁读缓存(cached=true)。
  - **雪崩(随机 TTL)**:有效答案 TTL = 基础 `ANSWER_TTL_SECONDS=2h` + `[0,ANSWER_TTL_JITTER_SECONDS=30min)` 随机抖动,避免大量缓存同时过期。
- **TTL 实现**:`RedisVectorStore` 不支持 TTL,故 `add()` 后对底层 key `aics:cache:{docId}` 单独 `EXPIRE`;过期后 RediSearch 惰性把它移出索引。
- **回显坑**:answer/sources/answered 这些 metadata 虽随 JSON 存了,但 RediSearch <b>只返回声明过的字段</b>;故在 `semanticCacheStore` 把它们声明为 TEXT(tenantId/kbId 为 TAG 用于过滤),否则命中回来 answer 是空的。
- **仅无历史单轮启用**:有多轮上下文时答案依赖历史,按「问题」做键会答错 —— `loadHistory` 非空则<b>跳过缓存读写</b>,直接走 RAG。
- **流式**:`POST /chat/stream` 命中缓存时按 code point 逐字回放缓存答案(保持前端流式 UX 一致),meta 事件带 `cached` 标记;未命中且持锁则在 `doneEvent` 回填缓存、`doFinally` 释放锁。
- **新增/改动**:`constant/CacheConstants`、`cache/SemanticCache`、`config/VectorStoreConfig`(+semanticCacheStore bean)、`constant/ChatConstants`(+NOT_FOUND_MARKER)、`dto/ChatResponseDTO`(+cached)、`service/impl/ChatServiceImpl`(召回前加缓存短路 + 三件套)。
- **验证(端到端全过)**:建 KB 传《售后政策.md》→ 同问两次:1st `cached=false` 2437ms、2nd `cached=true` 381ms 且<b>答案/溯源完整</b> → 换措辞仍命中 → 穿透:「你们有实体门店吗」`answered=false` 短 TTL≈118s → 雪崩:有效答案 key TTL≈7119/7702s 带抖动 → 击穿:5 并发同问仅 1 次 LLM → 流式:meta `cached=true`、逐字回放拼接正确 → <b>租户隔离:globex 问同一问题 cached=false 不串 acme 缓存;借 acme kbId 被「知识库不存在」拦</b>。验证后清理全部测试数据(缓存/历史/向量/DB)。
  > 决策:缓存与知识库分索引(独立 TTL);命中阈值 0.92 与召回 0.3 分离;answered 以答案兜底语判定(非 docs 是否为空);等待者廉价轮询锁释放(避免重复 embedding);语义缓存仅对无历史单轮启用。均与用户确认。
